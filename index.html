Is There a Better Way?

  • Manager gave me a paper to read and asked my opinion
  • It was on the idea of using a spreadsheet to define the steps of test automation
  • What would later become FIT
  • The intent is to allow non-programmers to contribute test cases

Pros

  • Tests don't just read like English, they are English
  • Makes it easy to share them with stakeholders
  • Facilitates getting contributions of tests from subject matter experts

Cons

  • It is its own programming language, albeit a very limited one: it has no support for cleanly adding loops, conditionals, or code reuse, unless you add that functionality yourself
  • It gives the false impression to contributors that whatever you type, it will work
  • There is no IDE support to guide test case creation
  • It's brittle: every change breaks a running test

Cons

  • The implementation of the logic is separated from the definition of the test, making it harder to wrap one's head around
  • Every unique bit of logic requires a unique bit of text to describe it, you can't readily rely on context
  • It promotes writing spaghetti test code
  • . . . and many more things I'm not fond of . . .
  • Which is why my recommendation was: find a happy medium

What I Learned
about Testing DSLs

  • Typical code is not suitable for domain experts
  • Spreadsheets and HTML documents are easy to read and write, but hard to reliably turn into working code
  • Especially if you need to add conditionals or loops or any other form of non-linear logic
  • Making test code easier to read and write is a good goal
  • But not at the expense of even rudimentary programming concepts, like branching, loops, and code reuse
Never give up Power
in return for Simplicity
You can and should have them both!

Real World Data

  • It is 2007, working at TransUnion

Equivalence Class Filtering

  • Solution was to do an intelligent filter on the data set
  • For every data type, wrote a translation function:
  • Names:
    'Smith' => 'Xxx', 'Jones' => 'Xxx'
    'McDonald' => 'XxXxx'
    'De La Joya' => 'Xx Xx Xxx'
  • Addresses:
    '123 N Oak Street' => '999 D Xxx St'
    'One Infinite Loop' => '9 Xxx Lp'

Building the Transforms

  1. Extract a histogram of all unique values in a field
  2. Analyze and look for patterns that are functionally equivalent
  3. Build an initial transformation function that transforms values matching that pattern into a single value, e.g. {token}
  4. Apply it and build a new histogram
  5. See what was not transformed by the function, and add logic to handle the next most common equivalence cases
  6. Repeat until every value has been transformed

Processing the Data

  • For every field, create a field=class feature-key
  • For every pair of fields, create a field=class+field=class feature-key
  • For every feature-key in the current record, test to see if we have already selected a record to represent that feature-key
  • If there are no new feature-keys in this record, skip it
  • Otherwise, link this record to every feature-key found in it

Selecting the Sample Set

  • I use a hash map to associate filter-keys to records
  • When a new record is added that includes filter-keys that we have seen previously, the record associated with those existing keys is changed to the new record
  • Earlier Records can get pushed out over time
  • Unique records will stay until the end
  • This means the algorithm favors records appearing later
  • Best to randomize the processing order to avoid bias

Minimum Sample Set

  • After processing all the records, the filter-key map contains pointers to the records that contain some kind of uniqueness about them
  • This sample set is a good approximation of the smallest possible set of records that contain every valid test case
  • At TransUnion, that sample set was 0.25%, or less than a million records out of 365+ million

Broken Tests

  • Found an issue where the API was not doing the right thing
  • After submitting a defect, had to figure out what to do with the test
  • Did not want to comment it out, lest I forget to bring it back later
  • Did not want to let it fail, as that would create test noise

What to do?

  • Created a mechanism for marking a test as "broken"
  • Defined the "expected" result
  • Also defined the actual, broken result
  • The test verifies the actual behavior, and if that passes, the test case is still broken, so the test is marked as "skipped"
  • Otherwise, the test fails, which means that the behavior of the application was changed, and the test case needs attention

What I Learned
about Test Runners

  • Unit testing frameworks are not built for handling this use case
  • Hiding the complexity of this conditional pass/fail/skip means adding some DSL to test scripts.
  • It is worth building that DSL, so that the test developer doesn't have to do much different when dealing with a broken test case vs. a normal, passing test case
  • This DSL also helps enforce test structure: Setup, Test, Verify, Teardown

Everything is OK

Failed means
Attention Required
If a test case fails, that means that what used to work no longer does, so the developers broke something, or the test case is not implemented correctly
Skipped means
it doesn't pass
but there is nothing we need to do about it right now
Yes, it's broken. We know. This is not the test you're looking for. Move along.

  • String fields, integer fields, boolean fields, required fields, optional fields, authenticated endpoint, create, retrieve, update, and delete tests, and so forth
  • When declaring a test class, implement all the interfaces that apply
  • The code will compile once all of the required test cases have been at least declared
  • Adding a test case is as simple as updating the interface and defining it everywhere that no longer compiles

What I Learned

  • It definitely pays to leverage the capabilities of the language for organizing the test code
  • The multiple-inheritance approach avoided duplicated code, but got too complicated
  • The interface approach works, but introduces a lot of boilerplate
  • I'm still looking for the right balance

Flagging Test Records

  • Lower environments are filled with everyone's test records
  • I built my automation to delete what it creates
  • But if a test fails, the records get left behind
  • Needed a way to tell my test records from everybody elses
  • So I could periodically programmatically delete them
  • That is when I decided on the naming convention:
    {prefix}.{timestamp}.{character-name}

What I Learned
about Naming Conventions

  • We brought on an outsourcing team to help implement the nearly 8,800 test cases I had defined
  • With so many people developing test cases, the test data from failed tests accumulated quickly
  • It became extra important to be able to programmatically delete records no longer needed
  • The importance was driven home when sometimes the outsourcing team didn't follow the conventions
Always follow
Naming Conventions
They make it easy to recognize which records were created when, why, and if it is okay to delete them now.

Framework Layers

  • Originally conceived of the testing DSL in layers
  • Abstraction Layer exposes an object-oriented interface for interacting with the records that the API lets you manage
  • The Interface layer exposes a functional interface to the underlying services
  • Tests would usually use the Abstraction Layer, except when it did not support what the test case demanded
  • My intentions were good: a multi-level abstraction of application interactions

What I Learned
about Layers

  • In hindsight, it wasn't a good idea
  • Most of the time, you used the high-level OOP DSL
  • Occasionally switching to using the low-level functions
  • It was jarring, and unpleasant
  • To make things worse, when we brought in the outsourcing team, they had a hard time knowing which to use, and often chose the wrong one
  • Eventually we added enough functionality to the Abstraction Layer that it was never necessary to go directly to the Interface Layer
One Testing DSL
Create a single, uniform approach to interacting with the application under test

Generating the DSL

  • The Keystone Admin testing framework consisted of three pieces: the Interface Layer, the Abstraction Layer, and the JSON Generator
  • One of each of these classes has to be created for every endpoint in the API
  • As there are around 30 endpoints, that's a lot of boilerplate
  • Decided the best way was to generate the classes programmatically

Generating the DSL

  • For each endpoint, crafted a JSON file describing its supported methods, urls, and request schema
  • Wrote a script to convert the JSON into the Java classes needed for the testing framework
  • Also, generated over 8,800 stubbed-out test cases
  • Got the generated code as correct as possible . . .
  • Then manually added one-off functionality that could not be generated

What I Learned
about Code Generation

  • Generating raw Java is much harder than generating code in a terse, custom DSL
  • Even so, definitely worth doing
  • Need a more polished way of doing code generation
  • Need to reorganize the generated code
  • Have other plans for improvement . . .

Next Generation

Not Java

  • Java's implementation of labda expressions has the unnecessary limitation of requiring all external variables referenced to be declared final or effectively final
  • This has the side-effect that using labda expressions for the code blocks (Setup/Test/Verify/Teardown) is restricted from initializing objects in the setup phase.
  • Other languages, most notably Scala, do not have this restriction
  • If at all possible, I will avoid Java in favor of a more elegant language

Putting It All Together

Automation Requirements

  • Automation can be launched from the command line
  • An environment variable controls which config file to read (e.g. local, DEV, PERF, STG, PROD, custom, etc.)
  • The framework reads the corresponding config file
  • Only environment-specific values are stored in config
  • Sensitive information is encrypted (esp. passwords)

Requirements - cont.

  • Test cases are tagged / assigned to groups
  • Sample groups include smoke, acceptance, functional, pedantic, regression, broken, etc.
  • Command-line parameter controls which group is executed
  • Tests that reveal a defect are tagged as broken
  • The test is also tagged with the defect number

Requirements - cont.

  • broken tests are not run, by default
  • broken tests are coded to have both an expected and an actual result, and to verify that the defect is still present
  • When run, if the behavior is still broken, the test case is marked as skipped
  • If the behavior has changed and the verification fails, the test fails, which means that manual verification is required to see if the ticket can be closed

Requirements - cont.

  • Once a broken test is fixed, the tag is switched to regression
  • The ticket number group is retained, however.
  • Every test logs the GIVEN/WHEN/THEN of what it is doing, so that the results log doubles as documentation of what is tested

Requirements - cont.

  • Test cases will never directly interact with the application under test
  • Test cases will always go through the abstraction layer
  • In the case of API testing, the abstraction layer provides a DSL for generating requests, and may provide some common compound request flows
  • In the case of UI testing, the abstraction layer abstracts windows, pages, features, processes, etc.

Requirements - cont.

  • Standard sets of test cases will be defined in interfaces
  • Test classes will implement the relevant interfaces
  • The code will compile when all the test cases are at least declared
  • When it makes sense to do so, use inheritance of test implementations and abstract methods to delegate the nitty-gritty details

Requirements - cont.

  • Unless business constraints prevent it, every functional test case will create test data during setup
  • Test records will (usually) follow a naming convention:
    {prefix}.{timestamp}.{character-name}
  • If the test case passes, the data created during setup will be deleted
  • If the test case fails, test records are left intact so that they may be inspected manually after the test run is complete

Requirements - cont.

  • Leftover test records will be periodically purged
  • Every test must be independent of every other test regardless of whether the testing framework allows for dependent tests
  • Functional tests must not depend on anything in the environment except an account with which to create its test data
  • Tests will be stand-alone - a separate test for every test condition - No optimizations, combining, stringing together, nothing
  • One possible exception to the above is for exhaustive testing that is 100% generated
